# visual-explain

複雑な説明を、単一の自己完結 HTML 資料として示すためのスキルである。資料化は常に読者の承認後に行い、正確さと判断材料を装飾より優先する。

## 役割とゲート

これは**説明専用**のスキルである。画面や UI を新規にデザインまたは改善する作業は対象外であり、環境にデザイン専用のスキルやツールがあればそちらへ譲る。デザイン成果物への承認判断を**説明する**資料は対象であり、成果物へのリンクを埋めてもよい。重複した提案はしない。

次の **3 条件をすべて**満たすときだけ、資料化を提案・実行する。

1. トポロジー、分岐する順序、状態変化、定量比較、before/after のいずれかが理解の核心である。
2. 誤解したときの損失または不可逆性という判断の重さが、資料化の手間と閲覧の摩擦を正当化する。
3. 1 文と少数の箇条書きでは、関係を保持できない。

**NO の例:** 単一主張、単純な要件確認、可逆な小変更、箇条書きで足りる列挙。迷ったらテキストを既定にする。UI の話題だから視覚的とは限らない。

自動的に候補を見つけた場合は、まず一言で「資料化すると判断しやすくなります。作成しますか？」と提案する。**明示的なユーザー承認の後だけ**生成する。明示起動は承認済みとして扱えるが、対象と型は確認する。description による自動起動はトリガーではなく best effort である。

## ワークフロー

承認後、次の 10 手順を順番に行う。

1. **ゲート判定:** 上の 3 条件を確認する。NO なら通常のテキスト説明へ戻り、資料は作らない。
2. **型を選ぶ:** 提案承認型、仕組み理解型、調査報告型から選び、対応する構成と図の契約を [references/patterns.md](references/patterns.md) で読む。
3. **動きを判定する:** 下の判定木で静的か、必要最小限の部品かを決める。
4. **図フォーマットを選ぶ:** `flow`、`layers`、`compare`、`matrix`、`timeline`、`kpi`、`bars`、`terms`、`details` の固定ライブラリを既定にする。座標計算、独自 CSS、独自 JavaScript は追加しない。
5. **構成する:** `assets/skeleton.html` を新しい資料へコピーし、`<!-- TITLE:BEGIN -->` と `<!-- TITLE:END -->` の間には非空のプレーンテキスト文書名を持つ `<title>` 要素を1つだけ置く。本文は `<!-- CONTENT:BEGIN -->` と `<!-- CONTENT:END -->` の間だけを編集する。ほかの領域は1バイトも変更しない。結論先行、必要時の用語表、型ごとの本文、必須の末尾節を入れる。描画規則は [references/design-system.md](references/design-system.md) に従う。
6. **機械チェックする:** `scripts/check.sh <絶対パス> --type <proposal|system|research>` を実行する。FAIL は修正して再実行し、成功するまで次へ進まない。
7. **目視セルフチェックする:** 下のリストを通し、機械検査だけで正しいと判断しない。
8. **保存する:** 下の保存規約に従い、衝突を避けて資料を保存する。
9. **開く:** `open-url "<絶対パス>"` を第一選択にする。なければ `open` または `xdg-open` を使う。起動の成功・失敗を問わず、資料の**絶対パスを必ず表示**する。GUI 表示は best effort であり、終了コード 0 でも表示を保証しない。
10. **ターミナルで要約する:** 資料の要点を 3 行、ファイルパス、そして「第一画面の判断ポイントと末尾のリスク節を読んでほしい」を表示する。

## 型と第一画面

### 提案承認型

第一画面は次の **3 要素だけ**に固定する。

1. 提案または推奨を 1 文。
2. **あなたが決めること**を 1 文。
3. 判断を左右する条件を最大 2 件。

同じ判断文を末尾に重複させない。第三者が第一画面を 3 秒見て「何の話か」「何を判断するか」を答えられなければ作り直す。新出用語が多いときだけ用語表を先に置き、現状と問題、等サイズで並べた before/after、代替案とトレードオフの比較を続ける。末尾の「リスクと弱い前提」「不確かな点」は省略しない。

### 仕組み理解型

TL;DR、必要時の用語表、全体地図、主要フロー、判断に不要な補足の `details`、末尾の「限界・確度」を置く。推論で補った箇所を明示する。

### 調査報告型

結論または推奨を最初に置く。主要な発見は 1 発見 1 ブロックとし、出典リンクを近傍に置く。定量データが必要な場合だけ可視化し、末尾に「限界・反証・確度」を置く。

## 図と説明の文法

固定するのはブロックの順序ではなく、次の制約である。

- 1 セクションは 1 問いに答える。目安は主張 1 行、根拠 2〜3 行。
- 図、表、短文のうち最も短く伝わる 1 つを主にする。図を使う前に、短文より明確になる理由を自問する。図は義務ではない。
- 根拠は主張または図の近傍に置く。離れた凡例やフッター凡例を置かない。
- `details` には判断に不要な検証過程と補足だけを入れる。核心、制約、反証を隠さない。
- 概ね 1 画面に収まるかを目視する。`.claim`、`.figure`、`.evidence`、`details.deep-dive` はこの制約のための部品であり、全セクションで使うものではない。

判断を変え得る主張には、対象の直近に **確認済み**、**推論**、**未確認** のいずれかをテキストで示す。出所と確度を混同せず、出所は局所注釈または節末参照に置く。未確認を確定事実として図示しない。注釈は hover 専用にせず、クリック、タップ、キーボードフォーカスでも開ける形にする。

ライブラリで表せない場合だけ、自由なインライン SVG を使える。使う直前に SVG を使う理由を HTML コメントで残し、描画規則に従うこと。弱いモデルはこの例外を使わず固定ライブラリ内で完結させる。同じ需要が繰り返されるなら、図フォーマットへの昇格を検討する。

## 動きの判定と部品契約

まず問う: **この説明の核心は、時間・順序・状態の変化そのものか。**

- **NO:** 静的な SVG または CSS の図で描く。アニメーションは使わない。
- **YES:** 固定部品ライブラリから最小の 1 つを選ぶ。
  - 手順・段階の進行: ステッパー。クリックで 1 段階ずつ進め、自動再生しない。
  - 状態遷移・データの流れ: ステッパーで図中ハイライトを段階移動する。
  - パラメータから結果への因果: v1 では `compare` で条件別に並置する。トグルとスライダーは実需要が出るまで追加しない。
  - 情報過多: `details` の折りたたみを使う。これは常時使用できる。

装飾目的の glow、pulse、常時ループは禁止する。読者のペースで進め、`prefers-reduced-motion` を尊重する。時間の部品はステッパーと `details` の 2 系統だけで、空間の図フォーマットと直交する。骨格内の部品ロジックは改変しない。

ステッパーには次を守らせる。

- クリック前に開始状態、終了状態、全ステップ数を予告表示する。
- 最終ステップ後に全段階を同時表示する「全体表示」を置く。
- ボタンは記号だけにせず、結果を予告するラベルにする。例: 「次へ: 設定を適用」。
- **無操作の不変条件:** 未到達ステップまたは未展開 `details` だけに、判断に必要な事実を置かない。操作は理解を深める追加経路であり、核心への唯一の経路ではない。

## 末尾節

提案承認型の末尾には、視覚的に区別した固定節を置く。

- **リスクと弱い前提:** 提案者自身が挙げる反証と弱点。空欄にしない。
- **不確かな点:** 未検証のことと仮定していること。

仕組み理解型と調査報告型では、対応する「限界・確度」または「限界・反証・確度」節を置く。理解度テスト、回答の強制、理解問題は置かない。このスキルの責務は、判断ポイント、リスク、確度を隠さない説明資料の提示までである。

## 目視チェック

機械チェックの後に、次を確認する。

- [ ] 第一画面だけで、第三者が 3 秒以内に話題と自分の判断を答えられる。
- [ ] ステッパーと `details` を一切操作しなくても、核心と判断材料に到達できる。
- [ ] 動き判定木に反するアニメーションがない。
- [ ] ラベルと注釈が対象の図から離れていない。
- [ ] 横スクロールがない。長い語は折り返され、表の列は増やしすぎていない。
- [ ] ノード、矢印、数値を原資料のコードまたは文書と逆照合した。
- [ ] 未確認事項を事実として描いていない。推論・未確認は対象の直近に直接表記した。
- [ ] 紫グラデーションや過剰装飾による generic な AI 見た目になっていない。

## 保存規約と縮退規則

git リポジトリ内では `<repo-root>/.visual-explain/`、リポジトリ外では `~/.visual-explain/` に保存する。ファイル名は `YYYY-MM-DD-<slug>.html` とする。同名があるときは上書きせず `-HHMM` を付ける。

リポジトリ内では `.gitignore` を編集しない。`git rev-parse --git-path info/exclude` で取得したファイルに `.visual-explain/` を追記してローカル除外する。書き込めない場合は、生成時にコミット対象に入らないよう注意を添える。自動削除はしない。生成時に 30 日超のファイルが 20 件を超えていたら一言知らせ、削除はユーザーに委ねる。恒久保存はユーザーが別の保管場所へ昇格する。

利用可能なモデルが固定ライブラリ、説明文法、正確性、無操作の不変条件を満たす資料を安定して作れない場合は、図を諦める。`matrix`、`terms`、または簡潔な文章へ縮退し、見栄えのために自由 SVG や動きを追加しない。縮退の具体的な適用基準は、別途の評価結果に従って確定する。

### Pi/Katsura Qwen 固有の保守的縮退

これは Pi 上の Katsura Qwen にだけ適用する、評価で確認した追加制約であり、他モデル向けの一般規則を緩めるものではない。

- 必須事実の因果は、原因→結果の文言を原文どおり保持するか引用する。言い換えで因果を補わない。
- 順序が明示的な必須事実でない限り、矢印や連番の図を描かない。
- 許される推論には対象の直近で **推論** と明記する。
- 因果・順序に確信がなければ、`matrix`、`terms`、または簡潔な文章へ縮退し、flow 図にしない。
